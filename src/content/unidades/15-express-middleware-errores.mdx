## ¿Qué es un middleware?

Un **middleware** es una función que se ubica **en el medio** del camino de la petición: recibe la información de una función, la transforma o valida, y se la pasa a la siguiente. Es la pieza central de Express.

Un middleware recibe **3 argumentos**:

```js
function miMiddleware(req, res, next) {
  // req  → la petición
  // res  → la respuesta
  // next → función que pasa el control al siguiente middleware
}
```

### El argumento next()

`next` es la clave. Termina el middleware actual de dos formas posibles:

<div class="proscons">
<div class="proscons__col pro"><p class="proscons__h">next() — todo bien</p><ul><li>Sin argumentos.</li><li>Le pasa el control al siguiente middleware/handler.</li></ul></div>
<div class="proscons__col con"><p class="proscons__h">next(error) — algo falló</p><ul><li>Con un objeto Error como argumento.</li><li>Salta directo al middleware de errores.</li></ul></div>
</div>

```js
function miMiddleware(req, res, next) {
  if (user.permisos !== 'admin') {
    return next(new Error('No tenés permisos para estar aquí'));
  }
  next(); // todo OK, sigue la cadena
}
```

### Cómo se montan

```js
// Global: se ejecuta ANTES de toda respuesta
app.use(miMiddleware);

// En una ruta puntual: sólo para ese endpoint
app.get('/', miMiddleware, (req, res) => { /* … */ });
```

<Callout type="warn" title="El orden importa (¡muchísimo!)">
Los middlewares se ejecutan **en el orden en que los definís**. Un middleware de autenticación tiene que ir **antes** del handler que protege. Y si olvidás llamar a `next()`, la petición **se queda colgada** para siempre.
</Callout>

## Manejo de errores

El objetivo es claro: que **ningún error tire abajo el servidor** y que todos queden **registrados** para analizarlos. En Express, el manejo de errores se hace con un middleware especial que tiene **4 parámetros** (`err` primero) y se define **al final** de todas las rutas:

```js
app.use(function (err, req, res, next) {
  console.error(err.stack);
  res.status(500).send('Tuvimos un problema, intentá más tarde.');
});
```

### Errores síncronos vs asíncronos

<div class="conceptos">
<article class="concepto"><div class="concepto__t">Síncronos</div><div class="concepto__d">Express los atrapa <strong>solo</strong>. Si un handler hace <code>throw</code>, se procesan automáticamente.</div></article>
<article class="concepto"><div class="concepto__t">Asíncronos</div><div class="concepto__d">Hay que pasarlos a mano con <code>next(error)</code> dentro de un <code>try/catch</code>, o el servidor cae.</div></article>
</div>

```js
// Síncrono: Express lo atrapa solo
app.get('/testerror', (req, res) => {
  throw new Error('error de prueba'); // capturado
});

// Asíncrono: hay que pasarlo a next() manualmente
router.get('/api/articulos-error', async (req, res, next) => {
  try {
    const items = await db.articulos.findAll({ attributes: ['NoExiste'] });
    res.json(items);
  } catch (error) {
    next(error); // sin esto, el servidor se cae
  }
});
```

<Callout type="tip" title="express-async-errors al rescate">
Para no escribir `try/catch` + `next(error)` en cada ruta async, se instala el paquete **`express-async-errors`**. Con sólo importarlo, captura automáticamente los errores de las funciones `async/await`.
</Callout>

## Excepciones no controladas

¿Y si un error escapa de todos los `try/catch`? Node emite eventos globales que **no evitan** la caída, pero permiten **registrar** la causa antes de morir:

<div class="deflist">
<div class="deflist__row"><div class="deflist__t">uncaughtException</div><div class="deflist__d">Se emite ante una excepción de JS no capturada en el ciclo de eventos.</div></div>
<div class="deflist__row"><div class="deflist__t">unhandledRejection</div><div class="deflist__d">Se emite cuando una Promesa es rechazada y nadie la atrapó con <code>.catch()</code>.</div></div>
</div>

```js
import process from 'node:process';

process.on('uncaughtException', (err) => {
  console.error('Error no manejado:', err);
  process.exit(1);
});

process.on('unhandledRejection', (error, promise) => {
  console.error('Promesa sin catch:', error);
});
```

## Gestores de procesos en producción

Como siempre existe la chance de que algo tire el servidor, en **producción** usamos un **gestor de procesos**: una capa extra que mantiene la app viva.

<div class="conceptos">
<article class="concepto"><div class="concepto__t">Reinicio automático</div><div class="concepto__d">Si la app se cae, la levanta sola.</div></article>
<article class="concepto"><div class="concepto__t">Monitoreo</div><div class="concepto__d">Métricas de rendimiento y consumo en tiempo real.</div></article>
<article class="concepto"><div class="concepto__t">Clúster</div><div class="concepto__d">Aprovecha varios núcleos del CPU.</div></article>
</div>

Los más conocidos: **PM2**, **Forever** y **StrongLoop**.

<Callout type="info" title="Azure y PM2">
Al publicar en **Azure App Service (Linux, stack Node)**, la plataforma ya provee **PM2** automáticamente. Vos sólo subís el código.
</Callout>

<Callout type="parcial">
- **Middleware** = función `(req, res, next)` que intercepta la petición.
- **`next()`** sigue la cadena · **`next(error)`** salta al manejador de errores.
- El **orden** de los middlewares define cuándo se ejecutan.
- Errores **síncronos** los atrapa Express; los **asíncronos** se pasan con `next(error)` (o `express-async-errors`).
- **`uncaughtException` / `unhandledRejection`** registran lo que escapó.
- En **producción**, un **gestor de procesos** (PM2) mantiene la app viva.
</Callout>
