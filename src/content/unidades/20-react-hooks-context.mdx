## Fragments: agrupar sin ensuciar el DOM

React exige **un único elemento raíz** por componente. Para agrupar varios hijos sin agregar un `<div>` extra al DOM, se usa un **Fragment**:

```jsx
function MiComponente() {
  return (
    <>
      <h1>Título</h1>
      <p>Contenido</p>
    </>
  );
}
```

<Callout type="tip" title="¿Por qué importa?">
Evita `<div>` innecesarios que pueden romper layouts (flex, grid) o estilos. La sintaxis corta `<>…</>` es la más usada.
</Callout>

## ¿Qué son los Hooks?

Los **Hooks** (React 16.8, 2018) son funciones que permiten **“engancharse”** al estado y al ciclo de vida desde componentes **funcionales**. Antes, eso solo se podía en clases.

<div class="conceptos cols-3">
<article class="concepto"><div class="concepto__t">useState</div><div class="concepto__d">Agrega estado local al componente.</div></article>
<article class="concepto"><div class="concepto__t">useEffect</div><div class="concepto__d">Ejecuta efectos secundarios (fetch, timers) tras el render.</div></article>
<article class="concepto"><div class="concepto__t">useContext</div><div class="concepto__d">Accede a un Context sin pasar props nivel por nivel.</div></article>
</div>

### Las 3 reglas de los Hooks

<ol class="pasos">
<li>Solo se llaman desde <strong>componentes funcionales</strong> o desde otros hooks (nunca en clases).</li>
<li>Solo en el <strong>nivel superior</strong>: nunca dentro de <code>if</code>, bucles o funciones anidadas.</li>
<li>Esto garantiza que se llamen <strong>en el mismo orden</strong> en cada render, para que React preserve bien el estado.</li>
</ol>

<Callout type="danger" title="El error más común">
Poner un hook dentro de un `if` o un `for`. Rompe el orden entre renders y React pierde la pista del estado. **Siempre arriba**, antes de cualquier `return`.
</Callout>

## useState: el estado

```jsx
import { useState } from 'react';

function Contador() {
  // [valor actual, función para actualizarlo] = useState(valor inicial)
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      Clicleado {count} veces
    </button>
  );
}
```

<div class="deflist">
<div class="deflist__row"><div class="deflist__t">count</div><div class="deflist__d">El valor actual del estado. React lo conserva entre renders.</div></div>
<div class="deflist__row"><div class="deflist__t">setCount</div><div class="deflist__d">Función para actualizar. Al llamarla, el componente <strong>re-renderiza</strong>.</div></div>
<div class="deflist__row"><div class="deflist__t">useState(0)</div><div class="deflist__d">El argumento es el valor inicial (solo se usa en el primer render).</div></div>
</div>

<Callout type="info" title="Es desestructuración de arrays">
`useState` devuelve un **array de 2 elementos**. Los nombres `count`/`setCount` los elegís vos; podés actualizar usando el valor previo: `setCount(prev => prev + 1)`.
</Callout>

Lo clave es el **ciclo**: un evento llama al `setter`, React actualiza el estado y **vuelve a renderizar** el componente con el nuevo valor.

```jsx
import { useEffect } from 'react';

// Solo al montar (1 vez): ideal para cargar datos iniciales
useEffect(() => {
  getDatos();
}, []);

// Cuando 'empresa' cambia
useEffect(() => {
  getResumen(empresa);
}, [empresa]);

// Limpieza al desmontar
useEffect(() => {
  return () => console.log('componente limpiado');
}, []);
```

## Comunicación entre componentes

<div class="proscons">
<div class="proscons__col pro"><p class="proscons__h">Padre → Hijo</p><ul><li>A través de <strong>props</strong>.</li><li>Los datos fluyen hacia abajo.</li></ul></div>
<div class="proscons__col con"><p class="proscons__h">Hijo → Padre</p><ul><li>El padre pasa una <strong>función callback</strong> como prop.</li><li>El hijo la invoca cuando algo pasa.</li></ul></div>
</div>

```jsx
// Hijo: invoca la función que recibió por props
const Nombre = (props) => <input onChange={props.onCambios} />;

// Padre: define la función y se la pasa al hijo
function App() {
  const onCambioHandler = (event) => console.log(event.target.value);
  return <Nombre onCambios={onCambioHandler} />;
}
```

## Renderizado condicional

Como en React todo es JavaScript, mostramos cosas según el estado de varias formas:

```jsx
// 1. if / variable de elemento
let boton = isLoggedIn ? <Logout /> : <Login />;

// 2. operador && (mostrar SOLO si la condición es true)
{unreadMessages.length > 0 && <h2>Tenés {unreadMessages.length} mensajes</h2>}

// 3. operador ternario (if-else en una línea)
<b>{isLoggedIn ? 'conectado' : 'desconectado'}</b>

// 4. devolver null para no renderizar nada
if (!props.warn) return null;
```

<Callout type="warn" title="Cuidado con && y números/strings vacíos">
`unreadMessages.length && <h2>…</h2>` renderiza un **`0`** si la longitud es 0 (porque `0` es falsy pero se imprime). Por eso conviene comparar: `length > 0 && …`.
</Callout>

## Renderizado de listas: map y key

Para renderizar arrays se usa `map()`, y **cada elemento necesita una `key` única**:

```jsx
const productos = [
  { id: 1, title: 'Cabbage' },
  { id: 2, title: 'Garlic' },
];

const items = productos.map((p) => <li key={p.id}>{p.title}</li>);
return <ul>{items}</ul>;
```

<Callout type="tip" title="¿Para qué sirve la key?">
La `key` (idealmente un ID de la base, **no** el índice) le permite a React identificar cada elemento entre renders, para saber qué insertar, borrar o reordenar sin recrear todo.
</Callout>

## Lifting State Up (levantar el estado)

Cuando **dos componentes hermanos** necesitan compartir el mismo dato, se **mueve el estado al ancestro común** más cercano, que se vuelve la **única fuente de verdad** y lo pasa hacia abajo por props.

```jsx
function MyApp() {
  const [count, setCount] = useState(0);     // estado en el padre
  const handleClick = () => setCount(count + 1);

  return (
    <>
      <MyButton count={count} onClick={handleClick} />
      <MyButton count={count} onClick={handleClick} />
    </>
  );
}

function MyButton({ count, onClick }) {
  return <button onClick={onClick}>Clicleado {count} veces</button>;
}
// Ahora ambos botones muestran y comparten el mismo número.
```

## Context: evitar el prop drilling

**Context** permite compartir datos “globales” (usuario, tema, idioma) a través del árbol **sin pasar props** en cada nivel intermedio.

```jsx
import { createContext, useContext } from 'react';

const ThemeContext = createContext('light');         // 1. crear

function App() {
  return (
    <ThemeContext.Provider value="dark"> {/* 2. proveer */}
      <ThemedButton />
    </ThemeContext.Provider>
  );
}

function ThemedButton() {
  const theme = useContext(ThemeContext);            // 3. consumir
  return <button>Tema: {theme}</button>;
}
```

<Callout type="warn" title="Usar con moderación">
Context es ideal para datos verdaderamente **globales**. Abusar de él dificulta la **reutilización** de componentes (quedan atados a un contexto). Para pasar datos entre pocos niveles, mejor props.
</Callout>

<Callout type="parcial">
- **Fragment** (`<>…</>`) agrupa hijos sin meter un `<div>` extra.
- **Hooks** dan estado/ciclo de vida a funciones. Reglas: solo en funciones y en el **nivel superior**.
- **useState** → `[valor, setValor] = useState(inicial)`; `setValor` re-renderiza.
- **useEffect**: `[]` = al montar · `[dep]` = cuando cambia · `return () =>` = limpieza.
- Padre→hijo por **props**; hijo→padre por **callback**.
- Condicional: `if`, `&&`, ternario, `return null`. Listas con `map` + **`key`** única.
- **Lifting state up**: el estado va al **padre común**.
- **Context** comparte datos globales sin **prop drilling**.
</Callout>
