## ¿Qué es Axios?

**Axios** es una librería basada en **promesas** para hacer peticiones HTTP, tanto en el **frontend** (React) como en el **backend** (Node). Es muy popular por su simplicidad y por funciones que `fetch` no trae de fábrica.

<div class="conceptos">
<article class="concepto"><div class="concepto__t">JSON automático</div><div class="concepto__d">Transforma la respuesta a JSON sin que llames a <code>.json()</code>.</div></article>
<article class="concepto"><div class="concepto__t">Errores claros</div><div class="concepto__d">Un status distinto de 2xx lanza error automáticamente.</div></article>
<article class="concepto"><div class="concepto__t">Interceptores</div><div class="concepto__d">Modificar peticiones/respuestas de forma centralizada (auth, logs).</div></article>
<article class="concepto"><div class="concepto__t">Config global</div><div class="concepto__d"><code>baseURL</code>, headers y timeouts compartidos por todas las llamadas.</div></article>
</div>

## Axios vs Fetch

| Característica | `fetch` (nativo) | `axios` (librería) |
|---|---|---|
| JSON automático | Manual (`.json()`) | Sí |
| Errores HTTP | No lanza en 4xx/5xx | Lanza solo |
| Cancelación | Experimental | Integrada |
| Interceptores | No | Sí |
| Sintaxis | Verbosa | Concisa |

<Callout type="info" title="¿Hace falta Axios?">
No: React funciona perfecto con `fetch` nativo. Pero Axios **ahorra código repetitivo** y resuelve cosas (errores, interceptores, baseURL) que con `fetch` tenés que armar a mano.
</Callout>

## Los métodos principales

```js
axios.get(url[, config]);        // leer
axios.post(url, data[, config]); // crear
axios.put(url, data[, config]);  // reemplazar
axios.delete(url[, config]);     // borrar
```

## Mismo ejemplo: Fetch vs Axios

Usando la API pública **JSONPlaceholder** (un backend falso para practicar). Primero con `fetch`:

```jsx
import { useEffect, useState } from 'react';

export default function ListaPosts() {
  const [posts, setPosts] = useState([]);

  useEffect(() => {
    fetch('https://jsonplaceholder.typicode.com/posts?userId=1')
      .then((res) => {
        if (!res.ok) throw new Error('Error en la carga');
        return res.json();            // hay que parsear a mano
      })
      .then(setPosts)
      .catch((err) => console.error(err));
  }, []);

  return <ul>{posts.map((p) => <li key={p.id}>{p.title}</li>)}</ul>;
}
```

Ahora con **Axios** (más limpio, con `baseURL`):

```jsx
import axios from 'axios';

const api = axios.create({ baseURL: 'https://jsonplaceholder.typicode.com' });

useEffect(() => {
  api.get('/posts', { params: { userId: 1 } })
    .then((response) => setPosts(response.data)) // ya viene parseado
    .catch((error) => console.error(error));
}, []);
```

<Callout type="warn" title="response.data, no response">
Axios envuelve la respuesta: los datos están en **`response.data`**, no en `response` directamente. Con `fetch` accedés tras `await res.json()`.
</Callout>

## Patrón de servicio (separar responsabilidades)

En vez de poner las llamadas HTTP dentro de los componentes, se **encapsulan en una capa de servicio**. El componente solo se preocupa por **mostrar**.

```js
// services/api.js — instancia reutilizable
import axios from 'axios';
export const api = axios.create({ baseURL: 'https://jsonplaceholder.typicode.com' });
```

```js
// services/postService.js — centraliza las operaciones del recurso
import { api } from './api';
const resource = '/posts';

export const postService = {
  getAllByUser: (userId) => api.get(resource, { params: { userId } }),
  create: (post) => api.post(resource, post),
  update: (id, post) => api.put(`${resource}/${id}`, post),
  remove: (id) => api.delete(`${resource}/${id}`),
};
```

```jsx
// App.jsx — el componente solo consume el servicio
import { postService } from './services/postService';

useEffect(() => {
  postService.getAllByUser(1)
    .then((res) => setPosts(res.data))
    .catch((err) => console.error(err));
}, []);
```

<Callout type="tip" title="Ventajas del patrón de servicio">
**Reutilización** (la misma lógica desde varios componentes), **separación de responsabilidades** (el componente solo muestra) y **mantenimiento/testing** más fáciles. Escala agregando `usuariosService.js`, etc.
</Callout>

## Interceptores

**Agregar el token de autenticación a todas las peticiones:**

```js
api.interceptors.request.use((config) => {
  config.headers.Authorization = `Bearer ${localStorage.getItem('authToken')}`;
  return config;
});
```

**Manejar errores de forma global:**

```js
api.interceptors.response.use(
  (response) => response,                 // si todo OK, pasa
  (error) => {
    console.error('Se produjo un error:', error.message);
    return Promise.reject(error);         // rechaza para manejarlo arriba
  }
);
```

<Callout type="analogia" title="Interceptores = control de aduana">
Toda petición que **sale** pasa por la aduana de salida (le ponen el sello del token); toda respuesta que **entra** pasa por la de llegada (revisan si hubo error). Lo escribís **una vez** y aplica a todo el tráfico.
</Callout>

<Callout type="parcial">
- **Axios** = librería de peticiones HTTP basada en promesas (front y back).
- Ventajas vs `fetch`: **JSON automático**, **lanza error** en 4xx/5xx, **interceptores**, `baseURL`.
- Los datos llegan en **`response.data`** (no en `response`).
- Métodos: `get`, `post`, `put`, `delete`.
- **Patrón de servicio**: encapsular las llamadas en `services/` → el componente solo muestra.
- **Interceptores**: lógica común a todas las peticiones (token) o respuestas (errores) en un solo lugar.
</Callout>
