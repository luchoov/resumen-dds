## ¿Qué es testing y por qué importa?

El **testing** es el proceso de **evaluar y verificar** que un software hace lo que se supone que debe hacer. Sus beneficios: **prevenir errores**, **reducir costos** de desarrollo y **mejorar el rendimiento**.

<Callout type="analogia" title="La red de seguridad">
Las pruebas son la **red del trapecista**. No evitan que te equivoques, pero hacen que equivocarte no sea catastrófico: si rompés algo que antes funcionaba, una prueba en rojo te avisa **al instante**, no el cliente en producción.
</Callout>

## Tipos de pruebas

A grandes rasgos, se dividen en dos grandes familias:

<div class="proscons">
<div class="proscons__col pro"><p class="proscons__h">Funcionales</p><ul><li>¿La app hace lo que el requisito pide?</li><li>Unitarias, integración, sistema, regresión, humo, aceptación…</li></ul></div>
<div class="proscons__col con"><p class="proscons__h">No funcionales</p><ul><li>¿Qué tan bien lo hace?</li><li>Rendimiento, carga, estrés, seguridad, usabilidad, escalabilidad…</li></ul></div>
</div>

### Pruebas funcionales (las más nombradas)

<div class="deflist">
<div class="deflist__row"><div class="deflist__t">Unitaria</div><div class="deflist__d">Prueba la unidad más pequeña (una función). Es de <strong>caja blanca</strong> y la más elemental.</div></div>
<div class="deflist__row"><div class="deflist__t">Integración</div><div class="deflist__d">Verifica que dos o más componentes interactúen bien entre sí (datos, llamadas a BD).</div></div>
<div class="deflist__row"><div class="deflist__t">Sistema</div><div class="deflist__d">Prueba todo el sistema como una unidad contra la especificación.</div></div>
<div class="deflist__row"><div class="deflist__t">Regresión</div><div class="deflist__d">Verifica que lo que <strong>antes</strong> funcionaba siga funcionando tras un cambio.</div></div>
<div class="deflist__row"><div class="deflist__t">Humo (smoke)</div><div class="deflist__d">Chequeo rápido de que las funciones <strong>básicas</strong> andan. ¿Prende y arranca?</div></div>
<div class="deflist__row"><div class="deflist__t">Aceptación</div><div class="deflist__d">Prueba formal contra los requisitos de <strong>negocio</strong>; replica al usuario real.</div></div>
</div>

### Manuales vs automatizadas

- **Manuales:** una persona ejecuta los pasos, hace clic, ingresa datos y valida resultados.
- **Automatizadas:** scripts y herramientas ejecutan los casos solos, registran resultados y defectos.

## TDD — Test-Driven Development

El **desarrollo guiado por pruebas** combina dos prácticas: **escribir la prueba primero** y **refactorizar**. La idea: traducir los requisitos a pruebas, de modo que cuando pasen, el software cumple lo pedido. Se resume en un ciclo de tres pasos:

<ol class="pasos">
<li><strong>Elegir un requisito</strong> de la lista (el que dé más conocimiento y sea simple).</li>
<li><strong>Escribir la prueba</strong> antes que el código (te obliga a pensar como el cliente).</li>
<li><strong>Verificar que falla</strong> (si no falla, el requisito ya estaba o la prueba está mal).</li>
<li><strong>Escribir el código mínimo</strong> que la haga pasar. <em>“Keep It Simple”.</em></li>
<li><strong>Ejecutar todas las pruebas</strong> y ver que pasan.</li>
<li><strong>Refactorizar</strong> para eliminar duplicación.</li>
<li><strong>Actualizar la lista</strong> de requisitos y repetir.</li>
</ol>

## Jest: el framework de pruebas

**Jest** es el framework de pruebas unitarias más popular en JavaScript (creado por Meta). Sus pruebas corren **en paralelo**, así que es rápido.

```bash
npm install --save-dev jest
```

Un primer test:

```js
// index.js
export function sumar(a, b) {
  return a + b;
}
```

```js
// suma.test.js
import { sumar } from './index.js';

test('suma dos números', () => {
  const resultado = sumar(1, 1);
  expect(resultado).toBe(2);
});
```

<div class="deflist">
<div class="deflist__row"><div class="deflist__t">test() / it()</div><div class="deflist__d">Define una prueba: una descripción + una función callback. <code>it()</code> es alias de <code>test()</code>.</div></div>
<div class="deflist__row"><div class="deflist__t">expect()</div><div class="deflist__d">Función de afirmación. Devuelve un objeto con “matchers” para comparar.</div></div>
<div class="deflist__row"><div class="deflist__t">describe()</div><div class="deflist__d">Agrupa varias pruebas relacionadas (por ej. del mismo recurso).</div></div>
</div>

### toBe vs toEqual (¡clásico de parcial!)

| Matcher | Compara | Úsalo para |
|---|---|---|
| `toBe()` | Identidad / valor primitivo (`Object.is`) | números, strings, booleanos |
| `toEqual()` | Igualdad **profunda** (recorre propiedades) | objetos y arrays |

```js
expect({ name: 'john' }).toEqual({ name: 'john' }); // PASA (igualdad profunda)
expect({ name: 'john' }).toBe({ name: 'john' });    // FALLA (distintos en memoria)
```

## Testing de backend con Supertest

**Supertest** permite probar **APIs HTTP** sin levantar el servidor a mano: le pasás tu app Express y dispara las peticiones. Se combina con Jest: Supertest **consume** los endpoints y Jest **evalúa** los resultados.

```bash
npm install supertest --save-dev
```

```js
import request from 'supertest';
import app from '../app.js';

describe('API usuarios', () => {
  it('responde la lista como array JSON', async () => {
    const res = await request(app).get('/api/usuarios');

    expect(res.statusCode).toEqual(200);
    expect(res.headers['content-type']).toMatch(/json/);
    expect(Array.isArray(res.body)).toBe(true);
    expect(res.body).toHaveLength(2);
  });
});
```

<Callout type="warn" title="No te quedes solo en las pruebas de humo">
Verificar únicamente status y headers son **pruebas de humo**: confirman que la API “está viva”, pero no que devuelve lo correcto. Hay que validar también el **cuerpo** de la respuesta, que es lo que el consumidor realmente espera.
</Callout>

## Alternativas y herramientas modernas

<div class="conceptos">
<article class="concepto"><div class="concepto__t">Vitest</div><div class="concepto__d">Alternativa moderna a Jest. ESM y TS nativos, API compatible, mucho más rápido. Ideal con Vite.</div></article>
<article class="concepto"><div class="concepto__t">MSW</div><div class="concepto__d">Mock Service Worker: intercepta peticiones HTTP a nivel de red para mocks realistas.</div></article>
<article class="concepto"><div class="concepto__t">React Testing Library</div><div class="concepto__d">Prueba componentes desde la mirada del <strong>usuario</strong>, no la implementación.</div></article>
<article class="concepto"><div class="concepto__t">Playwright / Cypress</div><div class="concepto__d">Pruebas End-to-End: flujos completos del usuario en un navegador real.</div></article>
</div>

### Estrategia recomendada (React + Express + Sequelize)

| Nivel | Herramientas | Qué prueba |
|---|---|---|
| Unitario | Jest / Vitest | Funciones de negocio, validaciones |
| Componentes | RTL + MSW | Componentes React aislados |
| Integración API | Jest + Supertest | Endpoints Express con Sequelize |
| End-to-End | Playwright / Cypress | Flujos completos del usuario |

<Callout type="tip" title="¿Cuánta cobertura?">
La recomendación es al menos un **80%** del código, priorizando siempre las **rutas críticas del negocio** (login, pagos, transacciones).
</Callout>

<Callout type="parcial">
- Testing = verificar que el software hace lo que debe (prevenir errores, bajar costos).
- **Funcionales** (¿qué hace?) vs **no funcionales** (¿qué tan bien?).
- La **unitaria** es la mínima (caja blanca, una función).
- **TDD**: rojo (prueba que falla) → verde (código mínimo) → refactor.
- **Jest**: `test/it`, `expect`, `describe`. **`toBe`** = primitivos · **`toEqual`** = objetos (profunda).
- **Supertest** prueba endpoints HTTP; combinado con Jest.
- **Vitest** = alternativa moderna (ESM/TS nativo, más rápido).
</Callout>
