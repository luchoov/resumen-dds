## El problema, en una pantalla

No es un CRUD genérico. El enunciado exige **reglas de dominio reales**:

- Una **tarea** pertenece **obligatoriamente** a un **proyecto**.
- El **responsable** debe figurar en los **integrantes** de ese proyecto.
- **Prioridad** y **estado** solo aceptan valores de una lista cerrada.
- Los **estados** avanzan por un flujo coherente (no se salta de cualquier estado a cualquier otro).
- Hay **roles** (colaborador / líder / admin) con permisos distintos.
- Cada cambio queda registrado en un **historial** (auditoría).

El sistema modela **4 entidades**: `usuarios`, `proyectos`, `tareas`, `historial_tareas`.

## Qué ENTRA (lo que pide el enunciado)

Esto es lo mínimo obligatorio. Si te preguntan, esto **sí o sí** lo tenés que saber ubicar en el código:

| Requisito | Dónde vive |
|---|---|
| Auth: registro, login, JWT, roles, sesión persistente | `auth.*` (back) + `AuthContext.jsx` (front) |
| Recurso principal con listado, filtros, detalle, alta, edición, acciones de estado | `tarea.*` (back) + páginas `Tarea*` (front) |
| Reglas de negocio **en servicios** del backend | `services/tarea.service.js` |
| Resumen administrativo con métricas | `tarea.service.js → resumen()` + `/resumen` |
| Historial de cambios visible | `historial.service.js` + detalle |
| Filtros, paginación y orden **resueltos en backend** | `tarea.service.js → listar()` |
| Middlewares: auth, autorización, validación, errores | `middlewares/` |
| Errores centralizados `(err, req, res, next)` | `error.middleware.js` |
| Pruebas con Jest + Supertest | `backend/tests/` |
| React Router con ruta comodín `*` | `App.jsx` |
| Capa Axios separada por recurso + interceptores | `frontend/src/api/` |

## Qué está "DE MÁS" (extras, no pedidos)

Acá está la trampa de estudio: **hay código que el enunciado NO pide**. Saber distinguirlo te hace quedar bien ("esto lo agregamos como decisión, no porque lo pidiera el enunciado"). Son tres:

<Callout type="tip" title="Los 3 extras de esta entrega">
**1. Tablero Kanban (drag & drop).** El enunciado pide un *listado con filtros* (tabla). El tablero por columnas es una **mejora de UX** — `TareaBoard.jsx`. La tabla con filtros/paginación/orden sí es lo pedido y también está.

**2. ABM de Proyectos.** El enunciado solo pide `GET /api/proyectos` (para llenar el select del formulario). Se agregó `POST` y `PUT /api/proyectos` **admin-only**, y la pantalla `/proyectos/nuevo`. Es un módulo análogo al de tareas, documentado como decisión.

**3. `GET /api/usuarios`.** No está en la lista de rutas del enunciado. Se agregó porque el formulario de tarea necesita los nombres de los posibles responsables. Devuelve datos no sensibles.
</Callout>

Si te preguntan por alguno, la respuesta de oro es: *"no lo pedía el enunciado; lo agregamos como decisión documentada en `docs/DECISIONES.md`, reusando el mismo patrón de capas"*. Eso demuestra que entendés el alcance.

<Callout type="analogia" title="La regla de oro de las capas">
La **route** es la recepcionista (te deriva). Los **middlewares** son seguridad (¿tenés entrada? ¿podés pasar a esta sala?). El **controller** es el mozo (toma el pedido, lo lleva). El **service** es la cocina (donde **realmente** se decide y se cocina). Los **models** son la despensa. El cliente nunca entra a la cocina.
</Callout>

Las relaciones se declaran con Sequelize en `models/index.js`: `Proyecto.hasMany(Tarea)`, `Tarea.belongsTo(Usuario, { as: 'responsable' })`, etc. Eso permite traer la tarea **con** su proyecto y su responsable en una sola consulta (`include`).

## El flujo estrella: crear una tarea, de punta a punta

Esta es la historia que tenés que poder contar entera. Seguila por las capas:

<ol class="pasos">
<li>**Front — formulario.** En `TareaForm.jsx`, React Hook Form valida que los campos no estén vacíos. El select de responsable muestra **solo** los integrantes del proyecto elegido (`watch('proyectoId')`).</li>
<li>**Front — service Axios.** Al enviar, `tareas.service.js → crearTarea()` hace `api.post('/tareas', tarea)`. El interceptor de `client.js` le agrega `Authorization: Bearer <token>` automáticamente.</li>
<li>**Back — route.** `POST /api/tareas` pasa por `authenticate` → `requireRole('lider','admin')` → `crearTareaRules` → `validate` → `controller.crear`.</li>
<li>**Back — controller.** `tarea.controller.js → crear()` llama `tareaService.crear(req.body, req.user)` dentro de un `try/catch` que hace `next(err)` si algo falla.</li>
<li>**Back — service (el corazón).** `tarea.service.js → crear()` valida: proyecto existe, no está finalizado/pausado, prioridad y estado válidos, **responsable integra el proyecto**. Si algo no cumple, lanza `AppError(400, ...)`.</li>
<li>**Back — persistencia + auditoría.** En una **transacción**, crea la tarea **y** registra el historial (`accion: 'creacion'`). O se guardan los dos, o ninguno.</li>
<li>**Vuelta.** El service devuelve la tarea serializada (con el flag `vencida`), el controller responde `201`, y el front muestra el éxito (o el error del backend si falló una regla).</li>
</ol>

<Callout type="warn" title="La pregunta clásica: ¿por qué la validación va en el service y no en el formulario?">
Porque el formulario **se saltea** con Postman o `curl`. La validación del front (React Hook Form) es solo para la experiencia del usuario. La **fuente de verdad** es el service del backend: ahí se valida contra la base (¿el responsable realmente integra el proyecto?), algo que el front no puede garantizar.
</Callout>
